## 核查 GLM-5.2-FP8 架构指标（serve_model_arch.py 输出审计）

用户贴出的四个指标（参数量 4541.22B / 激活 159.95B / KV cache 3744 MiB / FLOPs 319.9 GFLOPs）经核对全部来自 `serve_model_arch.py` line 1164-1184 的通用估算公式，且与公式输出逐一吻合，但**对该 MLA+MoE 架构均有误**：

- 真实总参数量 ≈ 755.6B（依据 safetensors.index.json `total_size=755,617,140,416` 字节，模型为 FP8 e4m3，权重约 1 字节/参数）。
- 真实激活 ≈ 54.6B；真实 FLOPs ≈ 109 GFLOPs/token；真实 KV cache ≈ 43.8 MiB(FP8)/87.7 MiB(BF16) 每 1K token。

### 三个根因 bug（在 serve_model_arch.py 估算段）
1. MoE 专家维度用错：FFN 单元取 `intermediate_size=12288`（稠密/共享专家维度）却 ×256 路由专家；真实 `moe_intermediate_size=2048`，比值 6×，导致总参数量被整体放大 6×。
2. KV cache 用 GQA 公式 `2·L·kv_heads·head_dim·2B`，把 MLA 当全量 GQA；MLA 实际只缓存压缩 latent `kv_lora(512)+rope_dim(64)=576`/层，比真值高 43–85×。最危险，会严重误导显存规划。
3. 激活参数未区分前 3 层稠密 FFN（`first_k_dense_replace=3`），假定全层都是 top-8/256 MoE。

### 修正已实施（serve_model_arch.py）
改动三处：
1. 新增 config 解析 `moe_intermediate_size`(→moe_ffn_hidden)、`n_shared_experts`、`first_k_dense_replace`。
2. 修正 `qk_nope_head_dim` 改为优先读 config（原来错误地用 head_dim-qk_rope=128，实际 config=192），并新增 `qk_head_dim`。
3. 重写估算段（原 line 1177-1198）：MLA 走低秩投影公式算注意力参数；路由/共享专家用 moe_intermediate_size；区分前 first_k_dense 稠密层；MLA 的 KV cache 只缓存压缩 latent (kv_lora+rope)*bf16。

修正后工具对 GLM-5.2-FP8 输出：**743.06B (active 39.98B) / KV cache 87.8 MB / 80.0 GFLOPs**。
关键修正：共享专家维度用 moe_intermediate_size=2048（非初判的 12288）；checkpoint 专家层为 layer 3–78 共 76 层，其中 layer 78 是 MTP 层，主干按 78 层(3稠密+75 MoE)计，与字节数 755.6GB 自洽。

回归验证：DeepSeek-V4-Pro(MLA) 1592.84B/68.69B、Mistral-Medium(dense) 125B、Keye-VL-30B-A3B active 2.63B(与命名 A3B 吻合，强旁证)。MiniMax-M3 config 字段命名不同返回空（非本次改动导致）。

### Phase 0 后端（serve_model_arch.py）
- `estimate_llm_metrics` 扩展：MoE/dense 分支重构为分别算 `routed_experts_total`/`shared_experts_total`/`dense_ffn_total`，保证 `sum(breakdown)==total_params`；返回值新增 `kv_bytes_per_token`(每 token 原始 KV 字节)、`hidden_size`、`num_layers`、`breakdown` 字典(attention/routed_experts/shared_experts/dense_ffn/embedding)。
